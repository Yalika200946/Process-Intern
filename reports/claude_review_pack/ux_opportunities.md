# UX opportunities (visual quality is largely already good - separate from the correctness gating in the classification CSV)

1. **Turn the review-mode banner into a real status system, not a static warning.** The existing orange "ENGINEERING REVIEW MODE" banner is a good pattern - extend it so every KPI card, chart, and table individually carries its own approval_status chip (VALIDATED / PROVISIONAL / EXPLORATORY / BLOCKED), matching `dashboard_readiness.csv`'s four levels, instead of one global disclaimer that a user can mentally tune out.
2. **Reuse the Evidence tab's honesty pattern everywhere.** The CIT model leaderboard ("แพ้ persistence ทุก fold") is the single best-designed piece of the product: plain language, a losing-is-shown-not-hidden verdict column, and a one-line "สรุปตรงๆ" takeaway. Every other logic area (fouling rate, forecast, network, furnace, economics) should get the same table shape.
3. **Make the two rankings' disagreement productive, not just disclosed.** Rather than a small tooltip admitting `engineering_priority_score` and `hx_ranking.json` disagree, show them side by side with the delta highlighted, so a reviewing engineer can see exactly where and why the two diverge - this is a real signal, not just a bug to hide.
4. **Fix the CIT-simulation chart's x-axis labels** (already flagged as a likely formatting bug in an earlier review this session: dates render out of chronological order, e.g. "Mar 27 → Jan 28 → Nov 28"). Low effort, currently undermines trust in an otherwise well-built chart.
5. **Give the per-HX custom multi-variable chart a second Y-axis or per-series normalization.** Overlaying `U_relative` (0-1 scale) with `Q duty` (thousands of kW) on one axis currently makes U_relative disappear - a real usability regression on an otherwise strong feature.
6. **Scope-limit UI affordances to match data scope.** If only 4 HX have an approved empirical reference, the HX picker/chart should visually distinguish those 4 from the other 12 (e.g. a badge), rather than presenting all 16 with identical chart chrome.
7. **Surface generation freshness prominently, not just in a header date string.** Given L01's finding that the whole dashboard is running on a frozen pre-MVP snapshot, a "data as of / pipeline generation" indicator with a staleness warning (e.g. "> 7 days since last canonical run") would have caught this discrepancy far earlier.
8. **Consider a dedicated "what changed since last generation" diff view** once the generation_id contract exists - useful both for engineers reviewing new pipeline runs and for catching accidental mixed-generation artifacts (CLI-007).
